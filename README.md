# Задание 1. Анализ и планирование

### 1. Описание функциональности монолитного приложения

**Управление отоплением:**

- Система поддерживает CRUD для работы с датчиками
- Пользователи могут включать/выключать отопление

**Мониторинг температуры:**

- Система сохраняет информацию о температуре в домах
- Пользователи могут просматривать температуру через web-интерфейс

### 2. Анализ архитектуры монолитного приложения

1. Язык программирования: Go
2. База данных: PostgreSQL
3. Архитектура: Монолитная, все компоненты системы (обработка запросов, бизнес-логика, работа с данными) находятся в рамках одного приложения
4. Взаимодействие: Синхронное, запросы обрабатываются последовательно
5. Масштабируемость: Ограничена, так как монолит сложно масштабировать по частям
6. Развертывание: Требует остановки всего приложения. Требуется выезд специалиста для подключение нового дома

### 3. Определение доменов и границы контекстов

Система "Умный дом":
Цель: Дать жителям возможность управлять температурой через приложение

1. **домен: Датчики**
	- **поддомен: Интеграции**
		- Обновление ПО и железа
		- Установка датчиков
	- **поддомен: Показания температуры**
		- Логирование данных о температуре
		- интеграция новых датчиков

2. **домен: Поддержка**
	- Ответы на вопросы клиентов

3. **Домен: Приложение**
	- Авторизация/регистрация
	- Разграничение прав пользователей
	- Отображение температуры дома/домов
	- Включение/Отключение отопления

4. **Домен: Маркетинг и Продажи**

### **4. Проблемы монолитного решения**

Для маленькой компании, которая обслуживает несколько домов, монолит - хорошее решение: просто поддерживать, все лежит в одном месте.  

С ростом количества домов и добавлением новых датчиков понадобится разбить систему на подсистемы. Пользователи должны иметь возможность самостоятельно настраивать систему.

# Задание 5. Работа с docker и docker-compose

Перейдите в apps.

Там находится приложение-монолит для работы с датчиками температуры. В README.md описано как запустить решение.

Вам нужно:

1) сделать простое приложение temperature-api на любом удобном для вас языке программирования, которое при запросе /temperature?location= будет отдавать рандомное значение температуры.

Locations - название комнаты, sensorId - идентификатор названия комнаты

```
	// If no location is provided, use a default based on sensor ID
	if location == "" {
		switch sensorID {
		case "1":
			location = "Living Room"
		case "2":
			location = "Bedroom"
		case "3":
			location = "Kitchen"
		default:
			location = "Unknown"
		}
	}

	// If no sensor ID is provided, generate one based on location
	if sensorID == "" {
		switch location {
		case "Living Room":
			sensorID = "1"
		case "Bedroom":
			sensorID = "2"
		case "Kitchen":
			sensorID = "3"
		default:
			sensorID = "0"
		}
	}
```

2) Приложение следует упаковать в Docker и добавить в docker-compose. Порт по умолчанию должен быть 8081

3) Кроме того для smart_home приложения требуется база данных - добавьте в docker-compose файл настройки для запуска postgres с указанием скрипта инициализации ./smart_home/init.sql

Для проверки можно использовать Postman коллекцию smarthome-api.postman_collection.json и вызвать:

- Create Sensor
- Get All Sensors

Должно при каждом вызове отображаться разное значение температуры
